Build a Next.js 14+ App Router project called "FastAccess" using TypeScript (strict mode) and Tailwind CSS. This is an accessible stadium navigation companion for fans with access needs (wheelchair/limited mobility, low vision, deaf/hard-of-hearing, sensory sensitivity).

IMPORTANT — SET UP TAILWIND CORRECTLY FROM THE START:
- Confirm globals.css (with @tailwind base/components/utilities) is imported in the root layout.tsx.
- Confirm tailwind.config content paths include src/**/*.{ts,tsx}.
- Confirm PostCSS is configured correctly.
- Verify styling actually renders before moving on — this must not end up as unstyled default HTML.

STRUCTURE — create exactly this layout:
- src/app/page.tsx — profile selector + route planner, Server Component shell
- src/app/api/route/route.ts — GET endpoint returning an accessible route for a given start/destination/profile
- src/app/api/forecast/route.ts — GET endpoint returning upcoming sensory events (noise/light) for a time window
- src/app/api/query/route.ts — POST endpoint, stub only (echo input for now)
- src/app/api/describe/route.ts — POST endpoint, stub only
- src/components/ProfileSelector.tsx — access-profile picker (see DESIGN SYSTEM below for how this should look)
- src/components/RouteMap.tsx — renders the computed path over a simple SVG venue map
- src/components/RouteNarration.tsx — Client Component, ARIA-live region (aria-live="polite") showing step-by-step directions
- src/components/SensoryTimeline.tsx — timeline list of upcoming noise/light events with intensity
- src/components/QuietZoneFinder.tsx — list of quiet rooms / sensory-friendly area with distance from user's section
- src/components/QueryBox.tsx — text input + submit, calls /api/query
- src/lib/venue/graph.ts — typed venue graph: 4 gates, 2 concourse junctions, 8 seating sections, 2 ramps, 2 elevators, 2 stairs-only routes, 2 quiet rooms, 1 sensory-friendly viewing area, 2 accessible restrooms, 1 service-animal relief area, guest services, first aid. Each edge has: distance (number), accessType ('step-free' | 'ramp' | 'elevator' | 'stairs-only'), noiseLevel ('low' | 'medium' | 'high'), lighting ('steady' | 'flashing').
- src/lib/venue/routing.ts — PURE FUNCTION: takes (fromNodeId, toNodeId, profiles: string[]) and returns the shortest eligible path, excluding edges that violate any selected profile's constraints (e.g., wheelchair profile excludes all 'stairs-only' edges). Use a simple weighted graph search (Dijkstra or BFS with weights).
- src/lib/sensory/schedule.ts — PURE FUNCTION: takes an elapsed match minute and returns any active/upcoming sensory events from a fixed schedule (anthem at -5, kickoff pyro at 0, goal events at fixed minutes e.g. 23 and 67, halftime show 45-60, final pyro at 90), each with intensity and type (noise/light).
- src/lib/ai/ — leave as clearly marked stub functions with TODO comments only. Do NOT call any AI API in this build. Real Claude tool-calling will be wired in separately, server-side only, never exposing any key to the client.

DESIGN SYSTEM — this app should look like accessible wayfinding/transit signage, not a generic SaaS dashboard. Ground every visual choice in this:

Fonts (use next/font/google):
- "Atkinson Hyperlegible" for all body and UI text. This typeface was designed by the Braille Institute specifically for low-vision legibility — use it everywhere, it's functional, not decorative.
- "IBM Plex Mono" for anything numeric or data-like: distances, times, minute markers, gate numbers. Use sparingly, only for that purpose.

Color tokens (add as custom colors in tailwind.config, use only these, no other colors):
- background: #FAFAF9 (near-white, not cream)
- ink: #14171A (primary text)
- accent-route: #1A4D8F (wayfinding blue — for step-free/route elements)
- accent-calm: #2F6B5E (sage/teal — for quiet zones and calm states)
- accent-caution: #C77D22 (warm amber — for sensory-intensity warnings; not red, to avoid alarm framing, but must hold 4.5:1 contrast on background)
- border: #E4E1DC (dividers, card outlines)
Every color must always be paired with an icon and a text label — never the sole carrier of meaning.

Layout:
- ProfileSelector: a grid of pictogram badge cards, not plain checkboxes — each access profile gets a bold icon in a colored square badge (evoking accessibility signage), the profile name, and its one-line effect description ("Excludes stairs-only routes"), stacked in the card. Selected state shows a visible border + checkmark, not just a color change.
- RouteMap / route result: render as a "station line" strip — each stop (gate, junction, ramp, section) as a labeled node connected by a line, in the signage aesthetic, rather than a paragraph or bullet list. Distance/time shown in the monospace face next to each node.
- SensoryTimeline: a departure-board-style list — time marker in monospace on the left, event name and intensity icon+label on the right, horizontal hairline rule between rows.
- Generous whitespace, bold hairline dividers between sections, minimal border-radius — flatter and more geometric than a typical rounded "SaaS card" look.

REQUIREMENTS:
- All venue and sensory data is static/deterministic, computed via pure functions — no database, no external services.
- Accessibility: hold this to a high bar since it's the core product — semantic HTML, full keyboard navigation with highly visible focus states (accent-route color, 3px minimum, offset from the element), respect prefers-reduced-motion (no decorative animation beyond simple opacity/state transitions), WCAG 2.1 AA contrast minimum on every text/background pairing.
- Testing: set up Vitest for unit tests (write tests for routing.ts covering each profile's exclusion rules, and schedule.ts covering event boundaries) and Playwright for one E2E test of selecting a profile and getting a route.
- Keep dependencies minimal — no UI kit or component library beyond Tailwind, no state management library beyond React hooks.
- Strict TypeScript, ESLint with zero warnings allowed.
- Do not add authentication, a database, or deployment config — this repo is submitted as source code only, not deployed live.
